✅ CONFIRMED: What matches exactly
================================================================

1. Ingestion via Celery
   Files: app/tasks/celery_app.py, app/tasks/ingestion_tasks.py, app/pipelines/ingestion_adapters.py

   - Celery is configured with Redis as broker/backend and a beat schedule running
     run_news_ingestion every 60 seconds.
   - RSSAdapter uses feedparser + httpx with exponential backoff via tenacity to fetch
     and normalize RSS feeds.
   - A WebScrapingAdapter also exists for sources without RSS.
   - Articles are deduplicated by a SHA-256 content_hash before saving to PostgreSQL.

----------------------------------------------------------------

2. TimescaleDB
   Files: docker-compose.yml, infra/docker/initdb/01_extensions.sh,
          app/models/event.py, app/models/market.py, app/models/gti.py

   - Docker image is timescale/timescaledb:latest-pg16. The init script runs
     CREATE EXTENSION IF NOT EXISTS timescaledb. The ORM models for events,
     market_data, gti_snapshots, and signals all carry
     {"timescaledb_hypertable": {"time_column_name": ...}} in their __table_args__.
     Fully confirmed end-to-end.

----------------------------------------------------------------

3. NLP Pipeline — DistilRoBERTa zero-shot
   File: app/pipelines/nlp_pipeline.py

   - Model is cross-encoder/nli-distilroberta-base, loaded lazily via
     transformers.pipeline("zero-shot-classification"). Two classification passes
     happen per article: a fast 3-class pass (normal/tension/escalation) and an
     expanded 11-category detailed pass.

----------------------------------------------------------------

4. VADER Sentiment
   File: app/pipelines/nlp_pipeline.py

   - vaderSentiment.SentimentIntensityAnalyzer is loaded lazily and used to produce
     a compound score -1…1. Confirmed exactly as described.

----------------------------------------------------------------

5. Severity Scoring
   File: app/pipelines/nlp_pipeline.py

   - Heuristic formula: 0.5 * class_weight + 0.3 * sentiment_factor + 0.2 * confidence.
     Each classification label carries a hardcoded weight
     (e.g. nuclear_threat → 0.95, military_escalation → 0.90). Exactly as claimed.

----------------------------------------------------------------

6. HDBSCAN Clustering
   File: app/pipelines/nlp_pipeline.py

   - EventClusterer uses hdbscan.HDBSCAN(min_cluster_size=3, metric="euclidean").
     Embeddings are generated by sentence-transformers (model: all-MiniLM-L6-v2),
     then clustered. Confirmed.

----------------------------------------------------------------

7. GTI with Exponential Decay
   File: app/pipelines/gti_engine.py

   - Formula: gti(t) = prev_gti * exp(-λ * Δt_hours) + Σ event_contributions.
     λ defaults to 0.05/hour (half-life ≈ 14h). Each contribution is weighted by
     severity * sentiment_factor * region_weight * decay. Confirmed exactly.

----------------------------------------------------------------

8. LightGBM as Primary Model
   File: app/pipelines/market_model.py

   - MarketImpactModel attempts to load/use lgb.LGBMClassifier. The model is
     serialized to model_artifacts/vol_spike_lgbm.pkl. Confirmed.

----------------------------------------------------------------

9. scikit-learn Fallback
   File: app/pipelines/market_model.py

   - If lightgbm cannot be imported (e.g., missing C++ toolchain on ARM Macs),
     the code falls back to sklearn.ensemble.GradientBoostingClassifier with
     identical hyperparameters. A separate Ridge regression model handles
     directional bias. Confirmed.

----------------------------------------------------------------

10. Euler-Maruyama SDE for GTI Simulation
    File: app/pipelines/simulators.py

    - ScenarioSimulator implements the SDE:
      dGTI = -λ * GTI * dt + μ_shock * dt + σ * dW
      with hourly time steps. The Wiener increment is rng.normal(0, sqrt(dt)).
      This is textbook Euler-Maruyama. Confirmed.

11. Monte Carlo for PnL Distributions
    File: app/pipelines/simulators.py

    - PortfolioSimulator.simulate() runs n_simulations=1000 draws from a
      multivariate normal distribution over asset returns. Outputs are the full
      (p05, p25, p50, p75, p95) PnL percentiles and a max_drawdown_estimate.
      Confirmed.

================================================================
⚠️ DISCREPANCIES: Where reality differs
================================================================

1. The surrogate ML model is trained on 100% synthetic data
   File: app/pipelines/market_model.py

   You implied this was a trained model. In reality, if no vol_spike_lgbm.pkl
   artifact is found on disk, the code auto-generates n=2000 synthetic training
   samples using rng.uniform(0, 1, ...) and trains on those. The label formula
   (0.2 * X[:,0] + 0.3 * X[:,7] + noise > 0.5) is a placeholder heuristic with
   no real historical grounding. There is no data pipeline that trains on actual
   market history.

----------------------------------------------------------------

2. The GTI SDE produces a single path, not a Monte Carlo ensemble
   File: app/pipelines/simulators.py

   You described "SDE simulations → Monte Carlo simulator." But ScenarioSimulator
   runs the SDE exactly ONCE per call, producing one deterministic-looking path.
   The Monte Carlo sampling (n_simulations=1000) happens only in the portfolio
   returns step, not on the GTI path itself. If you want a distribution of GTI
   futures, that is not implemented.

----------------------------------------------------------------

3. XGBoost is a dead dependency
   File: pyproject.toml

   xgboost==2.0.3 is listed in dependencies, but there is zero usage of XGBoost
   anywhere in app/. It is installed but completely unused.

----------------------------------------------------------------

4. TimescaleDB hypertables are never actually created via migrations
   File: alembic/versions/ (empty)

   The ORM models declare timescaledb_hypertable in their __table_args__, which is
   metadata-only annotation in SQLAlchemy — it does NOT cause
   SELECT create_hypertable(...) to be called. The alembic/versions/ folder is
   empty; there are no migration files. The timescaledb extension is only created
   by the Docker init script. Alembic will create standard Postgres tables without
   actually calling create_hypertable() on them. This is a real gap — your
   time-series tables are not actually TimescaleDB hypertables unless you manually
   run that call.

🔍 HIDDEN EXTRAS: Major things you didn't mention
================================================================

1. spaCy NER Pipeline
   File: app/pipelines/nlp_pipeline.py

   Model en_core_web_sm extracts named entities (GPE, ORG, PERSON, NORP, LOC,
   EVENT) from every article. Entity lists feed directly into the geo-risk vector
   and commodity/sector exposure calculations.

----------------------------------------------------------------

2. Sentence-Transformers Embeddings (all-MiniLM-L6-v2)
   Every processed event gets a 384-dim dense embedding stored as JSONB. These
   embeddings are what HDBSCAN actually clusters on — they are stored per-event in
   the database and are a major part of the pipeline.

----------------------------------------------------------------

3. Full Impact Graph Engine
   File: app/pipelines/impact_graph.py

   A BFS-based shock propagation graph connecting COUNTRY → COMMODITY → SECTOR →
   ASSET nodes with weighted edges. When a geopolitical event occurs, the shock
   propagates through the graph (with a damping/attenuation factor per hop) to
   score how strongly each asset is impacted. This is the backbone of the signal
   generator and is nowhere in your summary.

----------------------------------------------------------------

4. Signal Generator v2 with Reasoning Chains
   File: app/pipelines/signal_generator.py

   Signals include a full multi-step causal reasoning chain explaining how a
   geopolitical event → impact graph path → asset recommendation. Signals are far
   more than raw Buy/Sell/Hold outputs.

----------------------------------------------------------------

5. Backtesting Engine
   File: app/pipelines/backtesting.py

   A full backtest pipeline replaying historical events through the ML pipeline,
   computing Sharpe ratio, Calmar ratio, max drawdown, win/loss ratio, profit
   factor, signal-level breakdown, and equity curves. Not mentioned at all.

----------------------------------------------------------------

6. Real-time WebSocket Infrastructure
   File: app/core/websocket.py

   Multi-channel pub/sub WebSocket manager with channels for gti, signals, events,
   market, and alerts. The Celery beat schedule includes a 60-second
   broadcast_ws_updates task.
